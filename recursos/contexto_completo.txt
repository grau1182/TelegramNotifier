Conclusión sobre RatingKey='':
- NO es el bug principal; es comportamiento esperado para contenido nuevo fuera
  de plex_cache.json.
- Resolve-RatingKey (core/lib/cache-manager.ps1) solo devuelve valor si:
  1) DetectedMetadata.ratingKey (nunca se rellena en Process-Torrent), o
  2) Hay match en caché (exacto, alias o fuzzy >=85%).
- Kingsman no estaba en la caché (109 títulos en el log).

Causa directa del fallo: Plex devolvió 0 items en /search.

Causas probables (orden):
1. TIMING: qBittorrent ejecuta el script al completar la descarga; Plex puede no
   haber indexado G:\PELIS\ todavía. Todo en el log ocurre en el mismo segundo.
2. TÍTULO ES vs EN: búsqueda "Kingsman, El Servicio Secreto" vs Plex
   "Kingsman: The Secret Service". Minority Report sí funcionaba (título EN).
3. UNA SOLA QUERY: sin fallbacks progresivos (Kingsman solo, pre-coma, etc.).
4. AÑO no usado en la query (solo en scoring posterior, que no corre si 0 items).

Start-Sleep:
- En git (commit d81f0e0) existía en test/TelegramTorrent_Test.ps1 monolítico
  dentro de Initialize-PlexCache cuando no había fichero de caché.
- Eliminado en refactorización a core/lib/ (commit 33ef461).

2.4 UMBRALES DE SCORING (Test-PlexItemAcceptable)
-------------------------------------------------
  Score >= 100  → aceptar (match ruta exacta)
  Score >= 90   → aceptar
  Score >= 70   → aceptar (ej. Kingsman raíz +35 + año +40)
  Score >= 40 + año coincide + fuzzy combinado >= 60 → aceptar

Ejemplo Kingsman:
  Torrent: "Kingsman, El Servicio Secreto" (2014)
  Plex:    "Kingsman: The Secret Service" (2014)
  Raíz "Kingsman" == "Kingsman" → +35, año → +40 → 75 >= 70 ✓
